# State Handoff VM Cluster

Use this example when ODB networking is owned by a separate Terraform state and this state should create only a VM Cluster on an existing Cloud Exadata Infrastructure.

The pattern mirrors the OCI modules dependency style:

* A producer state, such as `examples/basic`, creates the ODB Network and ODB Subnets.
* The producer writes dependency JSON files with `output_path`.
* The consumer passes those JSON file paths into `gcp_odb_networks_dependency` and `gcp_odb_subnets_dependency`.
* The VM Cluster uses logical keys such as `primary`, `client`, and `backup` instead of copying full resource names into every cluster definition.

The module itself does not read remote state or object storage. Storage and transport for dependency files stays outside the module so the module remains backend-agnostic.

## Prerequisites

Before running it, confirm that:

* The networking stack has already created the ODB Network and ODB Subnets.
* Dependency JSON files are available to this root module.
* The existing Cloud Exadata Infrastructure is in the target region.
* Google provider authentication is configured with Application Default Credentials.
* The caller has permissions to create Oracle Database@Google Cloud VM clusters.
* If `db_server_ocids = null`, the caller also has permissions to read DB servers from the existing Cloud Exadata Infrastructure.

## Usage

```sh
cp terraform.tfvars.example terraform.tfvars
```

Edit `terraform.tfvars` with your project, region, dependency JSON paths, existing Exadata Infrastructure, VM Cluster settings, DB server OCIDs, and RSA SSH public key.

The `terraform.tfvars.example` file uses explicit placeholder `db_server_ocids` so local plans do not need to read DB servers from a real Google Cloud project. Replace them with one validated DB server OCID per VM. Set `db_server_ocids = null` only when the target project, API, and existing Cloud Exadata Infrastructure are real and Terraform may discover `AVAILABLE` DB servers.

The example includes local fixture dependency files for offline planning:

```hcl
gcp_odb_networks_dependency = "./dependencies/gcp_odb_networks_output.json"
gcp_odb_subnets_dependency  = "./dependencies/gcp_odb_subnets_output.json"
```

For production, replace those fixture files with JSON files generated by the networking stack or with equivalent maps assembled by your orchestration layer.

```sh
terraform init -backend=false
terraform validate
terraform plan
```

Review the plan carefully and stop there unless you are intentionally executing a real VM Cluster deployment.
